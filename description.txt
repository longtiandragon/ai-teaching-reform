7. passed 和 score 不要完全交给大模型

你说得很对：
不要让 AI 单独决定是否解锁下一步。

更稳的做法是：

最终得分 = 规则得分 × 60% + AI 评价得分 × 40%

或者：

最终 passed = 必选规则全部通过 && 综合分 >= 70

比如：

def decide_passed(rule_score: int, llm_score: int, critical_failed: bool) -> bool:
    final_score = int(rule_score * 0.6 + llm_score * 0.4)

    if critical_failed:
        return False

    return final_score >= 70

这样做有三个好处：

1. demo 现场更稳定
2. 不会被模型一句话误判
3. 更像真实教学评价系统

职业技能标准里本身也强调 JavaWeb 开发要能完成数据库、Spring MVC、JSON 交互、Spring/Spring MVC/MyBatis 整合开发等具体能力项，所以这些能力完全可以拆成可检查的 rubric，而不是只靠 AI 主观判断。

11. RubricEngine 要通用

你可以先支持这几类 criterion：

keyword          关键词命中
regex            正则检查
concept          概念覆盖
rag_grounded     是否引用/贴合资料
code_pattern     代码结构检查
api_design       接口设计检查
sql_design       数据库设计检查
llm_judge        大模型主观评价
anti_shortcut    防止直接整包生成

一个通用入口：

class RubricEngine:
    async def evaluate(
        self,
        student_input: str,
        rubrics: list[dict],
        task_context: dict,
        evidence: list[EvidenceChunk],
    ) -> tuple[list[RubricScore], int, bool]:
        scores: list[RubricScore] = []

        for rubric in rubrics:
            criterion_type = rubric["type"]

            if criterion_type == "keyword":
                score = self._check_keywords(student_input, rubric)

            elif criterion_type == "regex":
                score = self._check_regex(student_input, rubric)

            elif criterion_type == "concept":
                score = await self._check_concepts(student_input, rubric)

            elif criterion_type == "llm_judge":
                score = await self._llm_judge(student_input, rubric, task_context, evidence)

            elif criterion_type == "anti_shortcut":
                score = self._check_anti_shortcut(student_input, rubric)

            else:
                score = RubricScore(
                    criterion_id=rubric["id"],
                    name=rubric["name"],
                    passed=False,
                    score=0,
                    reason=f"暂不支持的评分类型：{criterion_type}"
                )

            scores.append(score)

        final_score = self._weighted_score(scores, rubrics)
        critical_failed = self._has_critical_failed(scores, rubrics)
        passed = final_score >= task_context.get("unlockPolicy", {}).get("minScore", 70) and not critical_failed

        return scores, final_score, passed

这样以后新增任务不改主流程，只加 rubric。

16. 你可以这样理解 LangGraph 在这里的价值

不要把 LangGraph 用成简单链式调用：

RAG → Prompt → LLM → 返回

那样体现不出你的项目优势。

你应该让 LangGraph 体现“教学决策流”：

学生是在问问题？
还是要提示？
还是提交作业？
还是要求 AI 直接给完整答案？
还是要解释报错？
还是要复盘学习？

不同意图走不同节点：

chat → 正常答疑
hint → 只给提示，不给完整答案
check → 严格按 rubric 验收
explain_error → 解释错误并要求学生修正
anti_shortcut → 拒绝整包代码，拆成当前任务
reflect → 生成学习总结

这样老师会感觉你的系统真的在“教学”，不是套了一个聊天框。

17. AI Prompt 要改成“课程运行时 Prompt”

不要写死“农宝系统”在系统 prompt 里。
系统 prompt 应该是通用的：

你是一个面向软件工程课程的 AI 实训助教。

你正在辅助学生完成一个项目式学习任务。
你不能一次性替学生完成完整项目。
你必须根据当前任务目标、评分标准、课程资料和学生提交内容进行引导。

教学原则：
1. 优先引导学生思考，而不是直接给完整答案。
2. 只围绕当前任务回答。
3. 对代码、SQL、接口设计要指出问题、原因和修改建议。
4. 如果学生要求生成完整项目或完整文件夹，要拒绝，并拆解为当前任务的下一步。
5. 反馈必须结合任务目标和评分标准。
6. 如果引用资料，请说明依据来自哪份资料。
7. 返回结构化 JSON，不要输出多余文本。

然后动态注入：

当前课程线：{course_line.title}
当前模块：{module.title}
当前任务：{task.title}
任务目标：{task.goal}
任务说明：{task.instruction}
评分标准：{rubrics}
RAG 检索资料：{evidence}
学生提交：{student_input}

这样就可以同时适配：

springboot-course-12
nongbo-admin-project
未来其他课程线


19. 你现在的核心卖点应该改成这 4 个

不要再说“炫酷页面”。
你要说：

1. 可配置课程线
不是写死 demo，支持多课程、多项目、多模块扩展。

2. AI 实训助教
不是直接生成代码，而是按任务目标和评分标准引导学生完成。

3. RAG 证据增强
回答和检查基于课程标准、项目需求、职业技能标准和学生历史提交。

4. 过程性评价
记录学生每次提交、AI 反馈、通过情况、卡点问题，教师端可查看学习分析。

这就是一个产品级方案。
