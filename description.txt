知识点拆解

无论具体题目是什么，一个典型RESTful后端的实现通常包含以下四个子任务，你可以按照这个思路来规划：

1. 定义接口路径与请求方式（Controller层）
- 确定URL、HTTP方法（GET/POST/PUT/DELETE）、请求参数和响应格式。

2. 编写业务逻辑（Service层）
- 处理数据校验、权限检查、调用持久层等。

3. 数据持久化（Mapper层）
- 使用MyBatis-Plus操作数据库，编写实体类、Mapper接口、XML映射或注解SQL。

4. 配置与调试（全局配置与测试）
- 配置数据源、事务、跨域；使用Postman或Swagger调试接口。

参考[2]中的技术方案，每一层都对应特定的技术组件，且数据流依次为：
浏览器 → Controller（接收请求） → Service（业务处理） → Mapper（数据库操作） → 返回结果

@RestController       // 1. 标记为REST控制器，自动返回JSON

public class AuthController {
    @PostMapping("/login")  // 3. 处理POST请求，路径为 /auth/login
    public Result login(@RequestBody LoginRequest request) {  // 4. 接收JSON请求体
        // 5. 调用Service层处理登录逻辑

    }
}

逐行解释：
- @RestController：将类标记为控制器，所有方法返回的数据自动转换为JSON。
- @RequestMapping("/auth")：设置该控制器下所有接口的公共路径前缀。
- @PostMapping("/login")：指定该方法处理 POST 类型的请求，路径为 /auth/login。
- Result login(...): 定义方法，返回统一结果对象 Result，便于前端统一处理。
- @RequestBody LoginRequest request：将请求体中的JSON自动绑定到 LoginRequest DTO中。
- 方法体内仅调用Service层，体现分层职责。

user_id int NOT NULL AUTO_INCREMENT COMMENT '用户ID'

- NOT NULL 表示该字段不允许为空
- AUTO_INCREMENT 表示字段值自动递增
- 在MySQL中，既有 NOT NULL 又带有 AUTO_INCREMENT 的字段，通常被定义为主键（常见惯例，课程资料未明确定义，但数据库设计通用规则如此）

知识点拆解

主键（Primary Key） —— 通俗理解：
主键是表中唯一标识每一行数据的“身份证号”。特征：
1. 不能为空（NOT NULL）—— 每个人都要有身份证。
2. 唯一不重复（通过 PRIMARY KEY 约束保证）—— 每个人身份证号不同。
3. 常用于自增整数（AUTO_INCREMENT）—— 系统自动给新数据分配一个不重复的号码。

在你的 test_user 表中，user_id 满足了这些特征，因此它就是主键。

局部示例与逐行解析

从资料中摘取 test_user 表的建表语句核心片段（仅展示主键相关部分）：

user_id int NOT NULL AUTO_INCREMENT COMMENT '用户ID'

- user_id：字段名，表示用户ID。
- int：字段类型是整数。
- NOT NULL：禁止为空，保证每行数据都有值。
- AUTO_INCREMENT：自增属性，插入新行时不指定值，系统自动加1。
- COMMENT '用户ID'：字段注释，帮助理解含义（实际建表时还会在最后加上 PRIMARY KEY (user_id) 明确声明为主键，但这段片段已足够推断主键身份）。
